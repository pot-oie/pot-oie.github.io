# 写作风格指南

这份文档基于 `src/content/blog/*.mdx` 中已有文章整理，用于之后扩写草稿、润色技术笔记、续写系列文章时保持 Pot 的原有文风。

## 整体语气

- 优先保留第一人称学习记录，不写成完全抽离的教科书口吻。
- 保留推理痕迹：从疑惑出发，写出错误直觉，再推进到更清晰的解释。
- 可以直接表达判断。“显而易见”“粗糙”“灾难性”“很丑”“亲切很多”这类词是自然质感的一部分。
- 文章应该像是在认真想明白一个问题，而不是宣布一个已经封装好的标准答案。

## 技术文章结构

技术文章通常适合按这个顺序展开：

1. **问题背景**：说明具体触发点。
2. **核心疑问**：写清楚哪里不符合直觉。
3. **直觉类比**：先用生活化比喻把机制讲出来。
4. **正式抽象**：在读者已经有直觉后，再引入公式、API、数据结构或系统术语。
5. **边界与取舍**：说明第一个想法为什么不完整、成本高，或者并不完全准确。
6. **沉淀结论**：用一句短结论收住文章。

好的补写应该补推理，不只是补定义。

## 标题习惯

- `##` 用于文章的大阶段。
- `###` 用于子问题、局部概念或方案分支。
- `####` 只在需要很小的证明、实现细节或注意事项时使用。
- 标题优先简洁，能用“训练集”“参数”“回归任务”说清楚时，不要写成解释型长标题。
- 少用“概念：解释”式标题，除非这个解释本身很有辨识度。
- 对比多个设计、拆多道题时，可以使用编号式标题。

## 段落和句式

- 大多数段落保持短到中等长度。
- 常用问题作转折，比如“为什么会这样？”“对吗？”“那问题来了：”。
- 保留个人化过渡，比如“一开始”“后来发现”“回头看”“现在问题就很简单了”。
- 学习型文章里可以保留不确定性，比如“这句话很粗糙”“后面再展开”“目前的理解是”。
- 不要把每一层推理都拆成标题。很多解释可以直接放在正文里，让文章读起来更像笔记，而不是大纲。
- 句子可以更干脆，少写“它更像是在回答……”这类重复模板。

## 格式偏好

- 用加粗标出核心概念，或者希望读者记住的一句话。
- 用行内代码标出标识符、路由、API、变量，以及 `input` / `output` 这类抽象占位。
- 用引用块写开头、提醒和短结论。
- 公式应该出现在解释之后，不要一上来就把抽象压给读者。
- 图片和交互组件应该紧跟它要解释的那段内容。

## 补写规则

之后填补或修改草稿时：

- 原文已有比喻且有效时，优先沿用这个比喻。
- 如果内容已经在已发布文章中完成收尾，后续文章只链接或简短承接，不整段搬运，避免系列内部重复。
- 如果内容来自用户自己写过但尚未发布的旧稿，优先迁移原文，再做必要裁剪、拼接和补缺；不要换一种说法重新造轮子。
- 先补例子，再补术语。
- 新增内容优先用具体场景托住，比如股价函数、贪吃蛇、AlphaGo Zero 这类例子，比抽象概括更适合当前风格。
- 不要把个人口吻磨平成百科式说明。
- 不要删掉自我修正、吐槽和带有学习痕迹的判断。
- 优先写“我现在怎么理解这个问题”，而不是“标准答案如下”。
- 如果某节本来是后续文章的坑，只补到当前文章需要的程度，给之后展开留空间。不要为了完整性强行补长。
- 收尾要短，点到核心问题即可。除非原文需要复盘，不要额外追加三点总结。
- 技术系列文章不需要每篇都强行承上启下。开头可以摘取核心句、提出一个具体疑问，或者简单说明文章内容；避免为了“引人”写营销式钩子。
- 不要默认用“下一篇要讲什么”“到这里先收住”这类教材式段落收尾。只有当文章本身需要导航时才写。

## 尽量避免

- 避免营销式开头、宏大口号和过度总结。
- 避免没有例子的术语堆叠。
- 避免把学习笔记包装得比实际完成度更完整。
- 避免用中性学术语言替换掉短促、有判断力的表达。
- 避免添加过长列表，除非文章本身就是清单或题解。
- 避免过度“教学化”的铺垫，比如连续解释“这是什么问题、它像什么、它说明什么”。能用一个例子讲明白时，就让例子自己工作。
